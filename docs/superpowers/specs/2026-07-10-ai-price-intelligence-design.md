# AI 商品公开链接比价后台设计

日期：2026-07-10

状态：已批准，进入实施

方案：标准实现版

## 1. 背景与目标

本项目面向单一博主管理员，用于调查公开网络中的 AI 数字商品商家。系统定时从 X 公开内容、Telegram 公共内容和已发现的公开店铺中寻找商品链接，先验证页面是否可访问并确实展示商品，再进入人工审核。只有审核通过且当前有效的商品才进入价格榜和货源榜。

核心输出不是对商家信誉背书，而是可核验的公开线索：具体商品页链接、页面标价、页面库存宣称、最后验证时间和证据置信度。自动发现的候选同时保存发现帖子链接；能确认店铺主页时再保存店铺主页链接。手工补链不要求虚构不存在的来源信息。

## 2. 已确认范围

### 包含

- 公网部署，后台全部页面必须登录。
- 系统中只允许一个管理员账号，不提供注册、邀请或新增管理员入口。
- 自动发现 X 和 Telegram 公共内容中的商品链接。
- 管理员可手工补充候选商品 URL。
- 候选链接验证、去重、商品信息抽取、规格归一化和人工审核。
- 商品、商家和来源链接直接展示并可安全打开。
- 同规格商品的价格榜和货源榜。
- 链接失效复检、采集任务状态和管理员审计记录。

### 不包含

- 公开访客页面、多管理员、角色权限或注册系统。
- 购买商品、提交订单、获取或验证账号凭据。
- 私人群组、私信、成员列表或非公开内容采集。
- 绕过登录、验证码、访问控制、反爬限制或平台频控。
- 价格历史图、网页证据截图、风险画像、到货提醒和报告导出；这些属于增强版。

## 3. 成功标准

1. 管理员可登录并看到候选审核、商家、商品规格、榜单、任务和设置页面。
2. 配置有效凭据且通过真实连接冒烟测试后，X/Telegram 连接器能在已获 API 权限、额度和平台可见范围内按关键词增量发现公开商品链接并保留来源 URL。
3. 缺少凭据、额度或权限时，连接器明确显示“未配置”或具体错误，不把失败展示成“没有结果”。
4. 只有公开匿名可访问、真实展示商品、审核通过且最近验证有效的链接进入榜单。
5. 不同规格、交付方式、共享属性、期限或计价单位的商品不会混排；价格榜按管理员选择的目标购买数量计算实际所需礼包和最低总支出。
6. 价格榜与货源榜相互独立，并显示计算依据、数据时间和置信度。
7. 每条榜单记录都可直接打开具体商品页；自动发现记录可追溯发现帖子，店铺主页仅在已确认时展示。

## 4. 技术架构

采用单仓库 TypeScript 架构：

- `web`：Next.js 16 管理后台、服务端接口和唯一管理员认证。
- `worker`：Node.js 后台进程，运行来源发现、协调隔离链接验证、商品抽取和周期复检任务。
- `validator`：无数据库和平台凭据的隔离抓取服务，只接收单个公网 URL 并返回结构化验证结果；普通 HTTP 与 Playwright 都在此服务内执行。
- `PostgreSQL`：业务数据、审核记录、链接检查和任务状态的唯一事实来源。
- `pg-boss`：基于 PostgreSQL 的持久化任务队列和定时调度，避免为标准版额外引入 Redis。
- `Playwright`：只在普通 HTTP 抽取无法处理公开动态页面时，由隔离 validator 降级使用。
- `Docker Compose`：本地和单台 VPS 部署 `web`、`worker`、`validator`、`postgres`；生产环境由反向代理提供 HTTPS。

数据流：

```text
定时任务
  -> X / Telegram 来源连接器
  -> 发现事件与候选 URL 去重
  -> worker 调用隔离 validator 完成安全链接验证
  -> 商品字段抽取与规格归一化
  -> 管理员审核
  -> 有效 Listing
  -> 价格榜 / 货源榜
```

web 与 worker 不共享内存，只通过数据库和任务队列通信。validator 不持有数据库、管理员、X 或 Telegram 凭据，只通过受限任务接口接收 URL 并返回结构化结果。采集连接器、链接验证器、商品解析器、归一化器和排名器均为独立模块，可用固定夹具单独测试。

## 5. 来源连接器

### 初始监控分类

系统只保留 AI 商品线索，初始关键词组覆盖：

- OpenAI / ChatGPT / GPT：Plus、Pro、Team、Business、API 额度、短期账号等公开商品表述。
- Anthropic / Claude：Pro、Team、API 额度和相关公开商品表述。
- Google / Gemini：Advanced、Google AI 套餐、Gemini API 和相关账号商品表述。
- Grok、Perplexity 及其他主流 AI 订阅或 API 套餐。
- K12、Bugteam、团队席位、教育资格等与 AI 服务商品共同出现的市场术语。

关键词、排除词和来源频道可在 `/jobs` 中启停和调整。与 AI 商品无关的候选在抽取阶段自动标为“不相关”，不进入审核主队列。关键词命中只代表调查线索，不代表商品合规或商家可信。

### X

- 使用官方 Recent Search API，查询包含商品关键词和链接的最近公开帖子。
- 用 `since_id` 增量读取，并优先使用帖子 `entities.urls.expanded_url`。
- 需要部署环境提供 X Developer App 的 Bearer Token 和相应额度。
- 标准接口通常只覆盖最近七天；系统必须在页面标明实际发现范围，不能声称“全网完整收录”。
- 401/403 时停用该来源并告警；429 按平台响应头退避；额度不可用时保留手工补链能力。

### Telegram

- Telegram Bot API 不支持全局搜索公共频道，因此不能用 Bot Token 冒充全网发现。
- 标准版使用专用用户账号的 MTProto 会话，通过 `messages.searchGlobal` 搜索公共消息，并对已知公共频道按消息 ID 增量扫描。
- 需要部署环境提供 `api_id`、`api_hash` 和加密后的专用 session；不得复用管理员个人账号。
- 遇到 `FLOOD_WAIT` 必须等待指定时间；用户态会话不可用时降级为种子公共频道和手工补链。

### 公开店铺

- 只验证来源连接器或管理员提供的候选 URL，不无边界遍历整个站点。
- 优先读取 JSON-LD `Product/Offer`，其次读取 OpenGraph 和可见 DOM。
- 普通请求无法解析公开动态页面时才使用隔离 Playwright。
- 403、robots.txt 禁止、验证码或登录墙都不绕过。新候选进入 `REVIEW_REQUIRED` 并记录具体原因；已有 Listing 按第 12 节分别进入 `RECHECK` 或 `INVALID`，并退出当前榜单。

## 6. 有效链接定义

商品链接必须同时满足：

1. 使用 HTTP 或 HTTPS，最终页面可匿名公开访问。
2. 最终响应为可接受的 HTML 页面，不是文件下载、停放域名、软 404、验证码页或登录墙。
3. 页面能识别商品标题，并至少能识别明确价格或有效购买动作。
4. 商品主题与发现关键词或人工选择的标准规格一致。
5. 管理员已审核通过。
6. 最近 24 小时内验证成功；超过时限立即退出当前榜单并等待复检。

系统保存原始 URL、规范 URL、每次重定向后的最终 URL、HTTP 状态、内容类型、页面指纹、验证耗时、失败原因和最后验证时间。URL 规范化仅移除 fragment 和明确的追踪参数，不能删除可能承载商品 ID 的查询参数。

## 7. 数据模型

- `AdminAccount`：固定主键 `id=1` 且数据库约束只允许这一行；保存唯一用户名、密码哈希、登录失败次数、锁定时间、会话版本、首次改密要求和改密时间。
- `AdminSession`：哈希后的会话令牌、到期时间、撤销时间和最近使用时间。
- `WatchSource`：平台、关键词或公共频道、启停状态、游标和最近运行结果；只保存凭据引用。
- `DiscoveryEvent`：公开来源 URL、来源平台、公共账号或频道、消息 ID、必要摘要、发布时间和发现时间；手工候选不创建伪造事件。
- `DiscoveryCandidate`：候选商品 URL、URL 指纹、`source_type`、可空的发现事件、抽取结果、审核状态和驳回原因；手工录入使用 `source_type=manual`。
- `Merchant`：名称、可空的店铺主页、平台、审核状态和最后验证时间。
- `ProductSpec`：标准品类与全部可比属性。
- `Listing`：商家、标准规格、原始/最终 URL、礼包单位、礼包数量、最低礼包数、原价、币种、折算价、库存证据、审核状态和链接状态。
- `ListingObservation`：每次验证时的原始价格、必付费用、税费状态、无条件优惠、币种、`fx_rate`、`fx_source`、`fx_observed_at`、库存宣称、页面指纹和时间；标准版用于榜单复算、新鲜度、稳定性和复检，不提供历史图表。
- `LinkCheck`：HTTP 状态、完整重定向链、最终 URL、页面判定、耗时、失败类别和检查时间。
- `CollectionRun`：任务起止时间、发现数、去重数、成功数、失败数和错误类别。
- `AuditEvent`：候选通过、驳回、合并、重抓、规格映射和设置变更。

## 8. 商品规格归一化

每个可排名商品必须生成完整的 `comparison_key`：

```text
产品商 + 产品线 + 套餐
+ 交付方式（账号/充值/邀请席位/API 额度）
+ 独享或共享 + 所有权
+ 地区或资格限制
+ 有效期 + 计费周期或承诺期
+ 配额类型
```

字段空值使用三种明确状态：`NOT_APPLICABLE` 表示该商品类型不适用，`UNDISCLOSED` 表示页面未披露，`PARSE_FAILED` 表示页面有内容但抽取失败。必填字段只有具体值或 `NOT_APPLICABLE` 时才算完整；出现 `UNDISCLOSED` 或 `PARSE_FAILED` 时进入“待归一化”，不得排名。

所有商品都必填：产品商、产品线、套餐、交付方式、地区限制、资格限制、计价单位和礼包数量。各模板额外必填：

- 订阅账号：独享或共享、所有权是否转移、有效期、计费或承诺期、席位数。
- 充值或邀请席位：有效期、计费或承诺期、席位数；所有权字段使用明确值或 `NOT_APPLICABLE`。
- API 额度：额度数值、额度单位、额度有效期和使用地区；席位字段为 `NOT_APPLICABLE`。
- 教育或 K12 类资格：资格要求、地区、有效期、席位数和所有权；不得把资格要求隐藏为普通账号。

售后期限是展示和筛选字段，不进入 `comparison_key`，避免因页面未披露售后而阻断基础比价。

礼包信息保存为 `bundle_qty`、`min_bundle_count` 和 `unit_basis`。只有 `comparison_key` 与 `unit_basis` 完全一致的商品才能进入同一个比较视图。价格榜要求管理员选择 `target_qty`，默认值为 1；系统计算满足目标数量所需的完整礼包数、实际购买数量、超买数量、最低总支出和有效单位价。这样不同礼包大小可以在同一计价单位下比较，但最低购买承诺始终可见且不会被单位价掩盖。

共享与独享、充值与交付账号、教育资格与普通账号、不同承诺期或计价单位严禁混排。自动抽取结果必须保存字段级置信度，低置信字段由管理员确认；管理员确认后规格置信度为 100%。库存单位不清楚时不能把“包数”擅自换算为“账号数”。

## 9. 价格榜

金额使用 Decimal 或整数最小币种单位，不使用浮点数。

```text
单礼包实付原币 = 标价 + 公开页明确的必付费用 + 公开页明确的必付税费 - 无条件优惠
所需礼包数 = max(min_bundle_count, ceil(target_qty / bundle_qty))
实际购买数量 = 所需礼包数 x bundle_qty
最低总支出人民币 = 单礼包实付原币 x 所需礼包数 x 抓取时汇率
有效单位价 = 最低总支出人民币 / 实际购买数量
```

主榜只纳入：审核通过、同一 `comparison_key`、同一 `unit_basis`、24 小时内链接有效、非缺货且价格可解析的商品。条件券、会员价、“低至”、价格区间或面议不进入主榜。系统不进入结算页，因此只采用公开商品页可核验的费用、税费和无条件优惠；页面声明存在必付费用但金额要到结算页才显示时，该商品标为“总价不完整”并退出主价格榜。

价格榜始终按有效单位价升序，不允许因其他因素把更贵商品排到更便宜商品之前。差距在 1% 内只添加“近似同价”标签，不改变顺序；仅在 Decimal 价格完全相同时，才依次按最近验证时间、字段置信度、近 30 天链接成功率和 Listing ID 排序，保证结果确定。

页面同时显示原始礼包价、币种、目标购买数量、实际购买数量、最低总支出、有效单位价、汇率来源和汇率时间。默认使用欧洲中央银行每日参考汇率折算法币，系统至少每日更新一次；汇率快照超过 24 小时或获取失败时，不参加跨币种排序，同币种比较仍可继续。USDT、USDC 等数字资产不自动等同于美元，只有管理员配置独立汇率来源后才参与跨币种排序。

## 10. 货源榜

货源榜只比较同一 `comparison_key`。它展示页面证据，不代表真实仓储或商家可信度。

### 明确库存数量

库存先换算为 `unit_basis` 对应的标准单位；单位不明时不能按明确数量处理。同一 `comparison_key` 在最近 30 天内，每个 Listing 只取最新一次正库存观察作为样本：样本数至少 5 时取 P95；不足 5 时使用当前样本最大值；参考库存始终至少为 10，以避免冷启动和零分母。

明确数量的商品得分范围为 80-100：

```text
参考库存 = max(10, 样本数 >= 5 ? P95 : 样本最大值)
数量分 = ln(1 + min(标准化库存量, 参考库存)) / ln(1 + 参考库存)
货源分 = 80 + 20 x 数量分
```

库存量必须大于 0；为 0 时按缺货处理并退出榜单。

### 没有库存数量但页面显示有货

推断得分范围为 20-79：

```text
有货证据 = 页面明确写“有货” ? 40 : 仅购买按钮可用 ? 20 : 0
新鲜度 = 20 x 2^(-距上次成功验证小时数 / 12)
链接稳定 = 15 x (近 30 天成功检查次数 + 1) / (近 30 天总检查次数 + 2)
商品广度 = 4 x min(1, ln(1 + 同产品族有效 Listing 数) / ln(21))
货源分 = 有货证据 + 新鲜度 + 链接稳定 + 商品广度
```

- 页面明确写“有货”：证据上限 0.69。
- 只有可用购买按钮：证据上限 0.49。
- 只有来源帖子宣称有货：不进入主货源榜，仅在候选详情显示，置信度上限 39%。
- 没有数量、状态文字或购买信号：显示“库存未知”，不进入货源榜。
- 缺货、预售或超过 24 小时未验证：不进入当前货源榜。

置信度使用确定公式：

```text
证据上限 = 明确数量 1.00 / 页面有货文字 0.69 / 仅购买按钮 0.49
连续一致系数 = 0.7 + 0.3 x min(连续相同库存状态次数 / 3, 1)
证据时效系数 = 2^(-距上次成功验证小时数 / 24)
置信度百分比 = round(100 x 证据上限 x 连续一致系数 x 证据时效系数)
```

管理员确认规格且链接当前有效是进入货源榜的前置条件，不再重复混入置信度。排序按货源分降序，再按置信度、最近验证时间和 Listing ID 排序。由于明确库存得分至少 80、推断得分最多 79，明确数量证据始终排在推断结果之前。

页面示例：`货源分 72 | 估算 | 置信度 61% | 页面宣称有货 | 2 小时前验证`。店铺商品数量、更新时间和链接稳定性只能作为辅助证据，不能显示成库存数量。

## 11. 页面与交互

- `/login`：唯一管理员登录，不提供注册；首次登录强制修改部署时设置的初始密码。
- `/dashboard`：价格榜、货源榜和最近发现；按标准规格、平台、币种和链接状态筛选。
- `/candidates`：显示发现帖子、店铺主页和具体商品页；支持通过、驳回、合并和重新抓取。
- `/merchants`：店铺主页、有效商品数、最近验证、来源平台和商品明细。
- `/specs`：维护标准规格，并处理低置信候选映射。
- `/jobs`：来源配置状态、任务游标、最近运行、限流或鉴权错误和手工补充 URL。
- `/settings`：修改密码、抓取周期、汇率和超时；外部 API 凭据只通过部署环境配置。

主界面使用固定列宽表格而不是瀑布流卡片。价格榜和货源榜为独立标签页。每行明确展示商品规格、商家、原价与参考价、货源证据、最后验证、审核状态和商品页链接；来源帖子与店铺主页可从详情打开。

复检、未审核、驳回、缺货和失效商品只能出现在对应管理队列，不出现在当前榜单。

## 12. 错误与生命周期

默认调度为：来源发现每 30 分钟一次；已上榜链接每 6 小时复检一次；榜单只接受 24 小时内的成功验证。管理员可调低频率，连接器必须在平台限额不足时自动延后，不能用高频重试消耗剩余额度。

候选状态机：

```text
DISCOVERED -> VALIDATING -> REVIEW_REQUIRED -> APPROVED -> 创建 Listing
                                  |-> REJECTED
VALIDATING -> RETRY_WAIT -> VALIDATING
```

Listing 状态机：

```text
ACTIVE -> OUT_OF_STOCK / INVALID / RECHECK / NEEDS_REVIEW
OUT_OF_STOCK -> ACTIVE（同域名、同商品且重新验证成功，可自动恢复）
RECHECK -> ACTIVE（仅瞬时故障恢复且商品指纹未变化时自动恢复）
INVALID / NEEDS_REVIEW -> ACTIVE（必须人工重新审核）
```

- 限流、鉴权失败、超时、DNS、TLS、验证码、登录墙、软 404 和字段缺失分别记录。
- X/Telegram 连接器的 429/5xx 按指数退避并保留游标；连接器鉴权 401/403 停用对应 `WatchSource` 并在后台告警，不改变已有 Listing。
- 商品页 404/410、软 404、登录墙或验证码页立即下榜并标为 `INVALID`；商品页 401/403、robots.txt 禁止或访问策略阻断时立即下榜进入 `RECHECK`，系统不尝试绕过。
- 超时、DNS、TLS 和临时 5xx 第一次失败时保留 `ACTIVE`，但必须显示上次成功时间；连续三次失败或距上次成功超过 24 小时时下榜并进入 `RECHECK`。
- 原域名或最终重定向域名变化时强制重新审核。
- 缺价格、规格不确定或页面只有联系方式时进入人工审核，不参与排名。
- 候选与任务使用唯一指纹和数据库租约，重试不得重复入库。
- 失效记录不自动删除，保留最小审计信息。只有同域名、同商品指纹的缺货或瞬时故障恢复可以自动回到 `ACTIVE`；域名、商品身份或关键规格变化必须由管理员重新审核。

## 13. 安全与隐私

- 密码使用 Argon2id；无默认密码；登录限流并在连续失败后短时锁定。
- 数据库为空时，启动程序才读取部署环境中的一次性管理员用户名和初始密码，写入 `id=1` 的 Argon2id 哈希；数据库已有记录时必须忽略初始凭据，重启不得重置密码。首次成功改密后记录 `bootstrap_password_consumed_at`，后续认证永不再比较环境初始密码。
- 会话 Cookie 使用 `HttpOnly`、`Secure`、`SameSite=Strict`；启用 CSRF 防护，改密后撤销全部旧会话。
- 验证器只允许 HTTP/HTTPS 公网 80/443。每次 DNS 解析和每跳重定向都阻断环回、私网、链路本地、CGNAT、组播、云元数据地址和 IPv4-mapped IPv6。
- 单次连接超时 5 秒、总超时 15 秒、最多 5 次跳转、HTML 上限 2 MB，并限制解压比例。
- validator 运行在独立容器中，不持有数据库或平台凭据，除受限任务接口的返回连接外无内网路由；浏览器无登录 Cookie、下载、剪贴板、摄像头或麦克风权限。
- 每域名初始并发 1、请求间隔至少 5 秒，尊重 `Retry-After`、robots.txt 和站点条款；403 或验证码不进行代理轮换或绕过。
- 远端文本只转义后展示，绝不渲染远端 HTML。
- 外链使用新窗口、`rel="noopener noreferrer"` 和 `Referrer-Policy: no-referrer`。
- 密码、X Token、Telegram session 和密钥只放部署环境或密钥服务，日志不得记录这些值或 URL 中疑似密钥的查询参数。
- 只保存公共消息 ID、公共账号或频道名、公开链接、时间和必要摘要；不采集手机号、成员列表或私人联系方式。

## 14. 测试与验收

### 单元测试

- URL 规范化、商品字段抽取、规格键生成、金额计算、汇率折算和两种排名算法。
- 登录锁定、会话撤销、CSRF、来源错误分类和链接生命周期。

### 集成测试

- 使用固定夹具验证 X/TG 发现、增量游标、去重和 429 重试。
- PostgreSQL 迁移、pg-boss 任务幂等、连续失败复检和恢复流程。
- 正常商品页、404、软 404、登录墙、验证码页和字段缺失页面判定。
- 私网 URL、DNS 重绑定、重定向至私网和超大响应必须被阻断。

### 端到端测试

- 唯一管理员登录、错误密码锁定、首次改密和旧会话失效。
- 候选发现、审核、规格映射、进入榜单、打开三类链接和失效下榜。
- 未审核、驳回、复检、缺货和失效商品均不出现在当前榜单。
- 不同规格绝不混排；并列价格、汇率不可用和库存推断有确定结果。
- 生产 HTTPS、数据库迁移、每日备份恢复和 worker 重启续跑完成一次演练。

## 15. 部署配置

生产环境至少提供：

- `DATABASE_URL`
- 初始管理员用户名和一次性初始密码
- 会话签名或加密密钥
- X Bearer Token（启用 X 发现时）
- Telegram `api_id`、`api_hash` 和加密 session（启用 Telegram 发现时）
- 汇率来源配置

缺少任一外部平台凭据时，只禁用对应连接器，后台其余功能和手工补链保持可用。部署检查必须验证 HTTPS、健康检查、数据库迁移、每日备份和日志脱敏。

启用 X 或 Telegram 连接器前，必须分别执行一次真实连接冒烟测试，记录当前账号的 API 权限、可见范围、额度或频控结果；测试失败时连接器保持禁用，不能仅凭环境变量存在就显示为可用。

## 16. 明确取舍

- 选择 PostgreSQL 持久任务队列，降低标准版运维复杂度。
- 选择“自动发现 + 人工审核”，不采用全自动直接入榜，以控制误报。
- 选择表格化后台，不采用参考截图中的瀑布流商品卡片，以提高横向比价效率。
- 货源分与商家信誉分严格分离；标准版不实现商家风险结论。
- 链接有效只证明页面在某时刻可访问并显示商品，不证明商品真实、库存真实或商家可信。
